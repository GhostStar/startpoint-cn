# gacha-faithful — eliminate C3032 (gacha rarity-mismatch crash)

C3032 = `ClientError(3032, "ガチャ結果レア度不一致...")`, thrown by the client's
`BallMovie.verifyResultBallRarity()` when the ball-drop animation computes a rarity
≠ the character's rarity. The server picks the gacha **seed**, so it must only emit
seeds for which the client computes the matching rarity.

## Root cause (two compounding bugs)

1. **`barAmulet` config was wrong** in `CN_GACHA_PHYSICS_CONFIG` (gacha-physics.ts):
   it was `{totalCount:5, lineCount:40, height:1}`; the real client AMF3 is
   `{totalCount:1, lineCount:5, height:40}` (decode it yourself with
   `amf3_decode.cjs`). Bar-amulet creation draws `chooseNumbers(0,lineCount-1,totalCount)`
   plus one `randomRangeFloat` per bar, all **before** the `playProbability` draw — so
   the wrong counts consumed the wrong number of MersenneTwister values and mis-aligned
   `playProbability` → wrong `moviePlayable`.

2. **The seed pools were generated by a too-crude model.** The shipped pools were
   labeled by a simplified physics (no pin-bounce solver / CCD), so a large fraction of
   seeds were mislabeled — the ★5 pools were ~95% wrong, which is the dominant C3032
   source.

## The fix

A **bit-exact, client-faithful** physics predictor — `world.cjs` (reuses
`native_hotpath.cjs`) — a literal transcription of the client's `gacha_physics` engine
(World.step two-pass integrate, ContactManager / persistent contacts, the
sequential-impulse `ContactPoint` solver, the narrowphase detectors, and the custom
`MathCompat` cos/sin/sqrt — no platform Math, so it is deterministic across runtimes).
`world.cjs.simulate(seed, movie)` is gated exactly like
`BallMovie.precalculateFieldResult` (runs physics only when `moviePlayable`, else uses
`initBallRarity`).

Validated against the real client (`assets/verified_seeds.json`, rarities captured from
actual C3032 events): **759/760 = 99.87%**; **100% on every seed where physics actually
runs** (the 1 miss is a `!moviePlayable` guarantee-pool edge that is a single
verified-pool data anomaly — a ★3 in a guarantee pool is mechanically impossible since
`ballStar4 ≈ 0`). A live stress test (validator emptied → server draws purely from the
filtered pools, ★5 included) produced **zero C3032**.

`filter_pools.cjs` then keeps only the seeds this predictor confirms, per pool slot.

## Files

- `world.cjs`, `native_hotpath.cjs` — the faithful predictor (bit-exact leaves verified
  by an independent transcription, 600k random cross-tests, 0 diff).
- `filter_pools.cjs` — filter `gacha_movie_seeds_<movie>.json` to client-faithful seeds.
- `validate.cjs` — measure the predictor vs `verified_seeds.json`.
- `amf3_decode.cjs` — decode the client gacha-movie `.amf3` config to verify values.

## Reproduce

```
node tools/gacha-faithful/validate.cjs      # -> 759/760 (99.87%)
node tools/gacha-faithful/filter_pools.cjs   # re-filter the pools (idempotent;
                                             #   pristine originals saved to
                                             #   assets/_pool_backup_prefilter/)
```

The pools in `assets/` have already been filtered. `gacha-physics.ts` `barAmulet` and
the per-seed debug log in `seed-validator.ts` (a flood/hang footgun when a pool is
exhausted) are fixed in this change.
